# STL

## vector как динамический массив, push_back и emplace_back

`std::vector` — динамический массив: непрерывный блок памяти, который может расти, в отличие от обычного массива фиксированного размера.

```cpp
std::vector<int> v;          // пустой вектор
std::vector<int> v2(10);     // 10 элементов, инициализированных нулями (int{})
std::vector<int> v3(10, 5);  // 10 элементов, каждый равен 5
std::vector<int> v4 = {1, 2, 3}; // инициализация списком
```

Доступ к элементам — через `[]` (без проверки границ, UB при выходе за пределы) или `at()` (с проверкой границ, бросает `std::out_of_range`):

```cpp
v4[0];     // 1, без проверки границ
v4.at(0);  // 1, с проверкой — бросит исключение при выходе за границы
```

### push_back

Добавляет элемент в конец вектора, принимая уже готовый объект — по значению, копируя или перемещая его в вектор:

```cpp
std::vector<std::string> v;
std::string s = "hello";
v.push_back(s);             // копирует s в конец вектора
v.push_back(std::move(s));   // перемещает s в конец вектора
v.push_back("world");        // создаёт временный std::string("world"), затем перемещает его в вектор
```

### emplace_back

Конструирует объект прямо внутри памяти вектора, передавая аргументы напрямую в конструктор элемента — без промежуточного временного объекта:

```cpp
std::vector<std::string> v;
v.emplace_back("hello"); // конструирует std::string("hello") напрямую в памяти вектора

v.emplace_back(5, 'a'); // конструирует std::string(5, 'a') = "aaaaa" прямо на месте
                          // push_back так не смог бы — пришлось бы сначала создать временный std::string(5, 'a')
```

`push_back` всегда требует готовый объект нужного типа, `emplace_back` принимает произвольный набор аргументов и сам вызывает конструктор элемента на месте. Для случаев с конструированием "на лету" `emplace_back` избегает лишнего перемещения временного объекта.

Реаллокация вектора (если `capacity` исчерпана) всё равно может потребовать перемещения/копирования уже существующих элементов в новый блок памяти — `emplace_back` не отменяет этого, он экономит только на конструировании нового добавляемого элемента.

## map со стандартным компаратором и особенность operator[]

`std::map<K, V>` — ассоциативный контейнер пар ключ-значение, отсортированных по ключу. По умолчанию использует `std::less<K>` как компаратор:

```cpp
std::map<std::string, int> ages;
ages["Alice"] = 30;
ages["Bob"] = 25;

for (const auto& [name, age] : ages) { // C++17 structured bindings
    std::cout << name << ": " << age << "\n"; // в алфавитном порядке: Alice, Bob
}
```

Реализован обычно как сбалансированное красно-чёрное дерево: вставка/поиск/удаление за O(log n), итерация всегда в отсортированном порядке ключей.

### Особенность operator[]: создаёт значение даже при чтении

Если ключ уже существует — возвращает ссылку на существующее значение. Если ключа нет — создаёт новый элемент с этим ключом и значением по умолчанию (`V{}`), и только потом возвращает ссылку на него:

```cpp
std::map<std::string, int> m;
std::cout << m["nonexistent"]; // выведет 0 — но попутно СОЗДАСТ элемент {"nonexistent", 0} в map

std::cout << m.size(); // 1 — хотя мы вроде бы просто "читали" значение
```

Даже просто прочитав значение через `[]` для несуществующего ключа, мы непреднамеренно модифицируем `map`. Из-за этого `operator[]` не существует в `const`-версии:

```cpp
void checkAge(const std::map<std::string, int>& ages, const std::string& name) {
    if (ages[name] > 0) { ... } // ошибка компиляции если ages — const
    // operator[] не может работать с const map, потому что потенциально модифицирует map
}
```

### Как правильно проверять наличие ключа без модификации

```cpp
std::map<std::string, int> ages = {{"Alice", 30}};

if (ages.find("Bob") != ages.end()) { // find НЕ создаёт элемент, безопасно для проверки
    std::cout << "Bob found\n";
} else {
    std::cout << "Bob not found\n"; // Bob не создаётся в map
}

// C++20
if (ages.contains("Bob")) { ... } // явная проверка наличия, без побочных эффектов
```

`find(key)` возвращает итератор на найденный элемент или `end()`, если ключа нет, и никогда не модифицирует контейнер.

## lower_bound / upper_bound

Алгоритмы (и одноимённые методы `map`/`set`), работающие на отсортированных диапазонах и находящие границы для значения с помощью бинарного поиска — O(log n) для методов контейнеров, O(log n) сравнений для алгоритмов из `<algorithm>`.

### lower_bound

Возвращает итератор на первый элемент, не меньший заданного значения (первый элемент `>= value`):

```cpp
std::vector<int> v = {1, 3, 3, 5, 7, 9};

auto it = std::lower_bound(v.begin(), v.end(), 3);
// первая тройка (индекс 1) — первый элемент >= 3

auto it2 = std::lower_bound(v.begin(), v.end(), 4);
// 4 в векторе нет, it2 указывает на 5 (индекс 3) — первый элемент >= 4

auto it3 = std::lower_bound(v.begin(), v.end(), 100);
// 100 больше всех элементов — it3 == v.end()
```

### upper_bound

Возвращает итератор на первый элемент, строго больший заданного значения (первый элемент `> value`):

```cpp
std::vector<int> v = {1, 3, 3, 5, 7, 9};

auto it = std::upper_bound(v.begin(), v.end(), 3);
// указывает на 5 (индекс 3) — первый элемент, строго больший 3

auto it2 = std::upper_bound(v.begin(), v.end(), 100);
// it2 == v.end() — нет элементов больше 100
```

### Параметры и сигнатура

```cpp
// алгоритм (свободная функция) — нужны итераторы и значение
template<class ForwardIt, class T>
ForwardIt lower_bound(ForwardIt first, ForwardIt last, const T& value);

// метод контейнера — только значение, контейнер уже знает свои границы
std::map<int, std::string> m = {{1, "a"}, {3, "b"}, {5, "c"}};
auto it = m.lower_bound(3); // работает по ключам map, возвращает итератор на пару {3, "b"}
```

Возвращаемое значение в обоих случаях — итератор; если подходящего элемента нет, возвращается `end()`, поэтому результат всегда нужно проверять перед разыменованием.

### Диапазон равных элементов

Разница между `lower_bound` и `upper_bound` для одного значения даёт диапазон всех элементов, равных этому значению (то же самое делает `equal_range` за один вызов):

```cpp
std::vector<int> v = {1, 3, 3, 3, 5, 7};

auto lo = std::lower_bound(v.begin(), v.end(), 3); // первая 3 (индекс 1)
auto hi = std::upper_bound(v.begin(), v.end(), 3); // первый элемент после троек — 5 (индекс 4)

std::cout << std::distance(lo, hi); // 3 — ровно столько троек в векторе
```

`lower_bound`/`upper_bound` как свободные функции из `<algorithm>` работают корректно только на уже отсортированном диапазоне — на неотсортированных данных результат логически некорректен (формально не UB в смысле краха программы, но гарантированно неверен с точки зрения алгоритма бинарного поиска). Методы `map`/`set` всегда корректны, потому что эти контейнеры поддерживают сортированность автоматически.
