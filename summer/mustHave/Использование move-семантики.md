# Использование move-семантики

## Эффективная инициализация полей из аргументов: по значению или по rvalue-ссылке

Классическая проблема: конструктор должен сохранить переданную строку как поле. Наивный вариант с `const&` всегда копирует:

```cpp
class Person {
    std::string name_;
public:
    Person(const std::string& name) : name_(name) {} // всегда копия, даже если аргумент временный
};

Person p("Alice"); // временная std::string("Alice") создаётся, затем КОПИРУЕТСЯ в name_ — лишняя работа
```

### Подход через rvalue-ссылку (&&)

Перегрузка, принимающая `T&&`, позволяет переместить, а не копировать, когда аргумент — временный объект:

```cpp
class Person {
    std::string name_;
public:
    Person(const std::string& name) : name_(name) {}        // копия — для lvalue
    Person(std::string&& name) : name_(std::move(name)) {}  // перемещение — для rvalue
};

Person p1("Alice");        // Person(std::string&&) — временная строка перемещается
std::string s = "Bob";
Person p2(s);                // Person(const std::string&) — s lvalue, копируется
Person p3(std::move(s));     // Person(std::string&&) — явно просим переместить s
```

Минус — приходится дублировать конструктор дважды, отличаясь только типом параметра и телом.

### Предпочтительный подход — принимать по значению

Чище — принимать параметр по значению и внутри один раз сделать `std::move` в поле:

```cpp
class Person {
    std::string name_;
public:
    Person(std::string name) : name_(std::move(name)) {} // один конструктор на оба случая
};

Person p1("Alice");       // временная строка строится прямо в параметре name, затем move в name_
std::string s = "Bob";
Person p2(s);               // s копируется в параметр name (lvalue), затем move в name_
Person p3(std::move(s));    // s перемещается в параметр name, затем move в name_
```

Если аргумент — rvalue, параметр `name` строится перемещением (дёшево), и его не жалко переместить ещё раз в `name_` — итог ноль лишних копий. Если аргумент — lvalue, копия неизбежна один раз (нельзя красть данные у объекта, которым кто-то ещё пользуется), но она происходит при построении параметра `name`, а `std::move(name) → name_` — это уже дешёвое перемещение. Итог: один конструктор без дублирования кода и минимально необходимое число копий/перемещений в обоих случаях.

## Moved-from состояние

После `std::move(obj)` и последующего перемещения объект `obj` остаётся в "valid but unspecified state" (валидном, но не определённом состоянии). Стандарт гарантирует только то, что объект остаётся корректным для своего типа (можно безопасно вызвать деструктор, присвоить новое значение), но не гарантирует, каким именно будет это состояние:

```cpp
std::vector<int> a = {1, 2, 3};
std::vector<int> b = std::move(a);

// a в valid but unspecified state
// НЕЛЬЗЯ полагаться на то, что a.size() == 0 — это деталь реализации, а не гарантия
a.push_back(10); // OK — безопасно, объект валиден, просто прежнее содержимое не определено
```

Само состояние не специфицировано стандартом (unspecified, не путать с UB — это разные категории), и на практике нет стандартного способа узнать, в каком конкретно состоянии находится объект после перемещения — кроме случаев, когда конкретный тип сам документирует своё moved-from поведение (например, `unique_ptr` специфицирует, что после перемещения становится `nullptr` — это часть его контракта, а не implementation detail).

### Пример ошибки

```cpp
void process(std::vector<int> data) {
    std::vector<int> backup = std::move(data); // data теперь в unspecified состоянии

    if (!data.empty()) {      // нельзя полагаться: может быть true или false — не специфицировано
        data[0] = 100;         // потенциально UB, если data всё-таки пуст
    }
}
```

Корректная модель: после `std::move(obj)` следует относиться к `obj` как к новой, неинициализированной в смысловом плане переменной того же типа — с ней можно безопасно работать (присвоить новое значение, уничтожить), но нельзя предполагать что-либо о её текущем содержимом.

## Отсутствие необходимости move из результата функции, возвращённого по значению

Частая ошибка — оборачивать возвращаемое значение в `std::move`:

```cpp
std::vector<int> createVector() {
    std::vector<int> result = {1, 2, 3};
    return std::move(result); // антипаттерн — не нужно и может быть вредно
}
```

Это мешает copy elision (и NRVO). Компилятор и без `std::move` либо вообще не создаёт копию (строит `result` сразу в месте назначения — настоящий copy elision), либо, если это невозможно, автоматически рассматривает локальную переменную как rvalue при `return` и применяет перемещение неявно:

```cpp
std::vector<int> createVector() {
    std::vector<int> result = {1, 2, 3};
    return result; // правильно — компилятор сам elides копию либо неявно делает move
}
```

Явный `std::move` здесь может помешать настоящему copy elision (когда объект строится один раз сразу в финальном месте), заставив компилятор вместо этого использовать менее эффективный move-конструктор. Правило: никогда не оборачивай локальную переменную в `std::move` при `return`.

## Необходимость move для явной передачи владения unique_ptr

`std::unique_ptr` некопируем (конструктор копирования и `operator=` удалены — `= delete`), потому что моделирует единоличное владение ресурсом: если бы копирование было разрешено, два `unique_ptr` указывали бы на один объект, и оба попытались бы его удалить — double free.

```cpp
std::unique_ptr<int> p1 = std::make_unique<int>(5);
std::unique_ptr<int> p2 = p1; // ошибка компиляции — конструктор копирования удалён
```

Единственный способ передать владение — переместить:

```cpp
std::unique_ptr<int> p1 = std::make_unique<int>(5);
std::unique_ptr<int> p2 = std::move(p1); // OK — владение передано от p1 к p2

// p1 теперь nullptr — это специфицированное поведение именно для unique_ptr
// p2 владеет ресурсом
```

`std::move(p1)` явно сигнализирует "я больше не буду использовать p1 как владельца, можно забрать у него владение" — без этого сигнала компилятор не может автоматически переместить `p1`, потому что это именованная переменная (lvalue), которая потенциально могла бы использоваться и дальше.

Это касается и передачи в функции:

```cpp
void takeOwnership(std::unique_ptr<int> p) { /* p владеет ресурсом внутри функции */ }

std::unique_ptr<int> ptr = std::make_unique<int>(5);
takeOwnership(ptr);             // ошибка — нельзя скопировать
takeOwnership(std::move(ptr));  // OK — явно передаём владение, ptr становится nullptr
```

## Почему std::move не выполняет никакого кода

`std::move` не перемещает ничего сам по себе. Он не вызывает move-конструктор, не меняет состояние объекта, не выполняет никакой рантайм-логики — это, по сути, просто `static_cast` к rvalue-ссылке:

```cpp
// упрощённая реализация std::move
template<typename T>
constexpr std::remove_reference_t<T>&& move(T&& t) noexcept {
    return static_cast<std::remove_reference_t<T>&&>(t);
}
```

Это функция, которая меняет только то, как компилятор смотрит на тип выражения — "перекрашивает" lvalue в rvalue-ссылку, сообщая компилятору: при разрешении перегрузок рассматривай это выражение как временный объект, можно выбирать move-перегрузки конструкторов/операторов.

```cpp
std::string s = "hello";
std::move(s); // само по себе ничего не делает — s не меняется, перемещения не произошло

std::string s2 = std::move(s); // здесь происходит реальное перемещение —
                                  // не потому что std::move что-то "сделал",
                                  // а потому что move-конструктор std::string выбран
                                  // на основе того, что std::move(s) теперь имеет тип rvalue-ссылки
```

Реальная работа (переприсвоение внутренних указателей, обнуление source-объекта) выполняется внутри move-конструктора или move-присваивания того типа, к которому применили `std::move`, а не самим `std::move`. Если у типа нет move-конструктора, `std::move` приведёт к выбору обычной copy-перегрузки (`T&&` неявно конвертируется в `const T&` при отсутствии подходящей `T&&`-перегрузки) — `std::move` сам по себе никогда не гарантирует, что перемещение реально произойдёт, он лишь открывает компилятору такую опцию при разрешении перегрузки:

```cpp
struct Simple { int x; }; // нет явного move-конструктора, нет ресурсов

Simple a{5};
Simple b = std::move(a); // std::move(a) — rvalue-ссылка, но сгенерированный
                            // тривиальный move-конструктор просто копирует int —
                            // поведенчески неотличимо от копирования
```
