# Переменные

## Статическая типизация

C++ — язык со статической типизацией: тип каждой переменной известен и фиксирован на этапе компиляции, а не во время выполнения программы. Компилятор заранее знает, сколько памяти выделить под переменную, какие операции с ней допустимы, и может проверить корректность типов до запуска программы:

```cpp
int x = 5;
x = "hello"; // ошибка компиляции — тип x зафиксирован как int
```

В языках с динамической типизацией (например, Python) тип привязан не к переменной, а к значению, и может меняться в рантайме — аналогичный код там был бы валиден. Статическая типизация даёт два практических преимущества: ошибки несоответствия типов ловятся на этапе компиляции, а не в рантайме у пользователя, и компилятор может генерировать более эффективный машинный код, заранее зная размеры и представление данных.

## Тип переменной

Тип переменной определяет: какой набор значений она может принимать, сколько байт памяти занимает, какие операции над ней определены, и как интерпретируется её битовое представление в памяти:

```cpp
int a = 5;       // целое число
double b = 5.0;  // число с плавающей точкой
char c = 'A';    // один символ (по сути, маленькое целое)
```

Тип — это не просто ярлык, а инструкция компилятору, как обращаться с битами в памяти: одна и та же последовательность бит интерпретируется по-разному в зависимости от типа (целое со знаком, число с плавающей точкой, символ).

## Размер переменной и размер типа

Размер типа — количество байт, которое занимает в памяти любой объект этого типа. Узнаётся оператором `sizeof`:

```cpp
sizeof(int);  // размер типа int
sizeof(x);    // размер типа переменной x
```

`sizeof` — оператор времени компиляции, а не функция, выполняющаяся в рантайме. Значение вычисляется компилятором на основе типа выражения ещё до запуска программы — `sizeof(x)` не "измеряет" переменную в рантайме, компилятор просто подставляет константу:

```cpp
int arr[10];
sizeof(arr);                   // 40 на типичной системе (10 * sizeof(int))
sizeof(arr) / sizeof(arr[0]);  // классический способ узнать число элементов массива
```

Размер переменной равен размеру её типа. Стандарт C++ не фиксирует конкретные числовые значения размеров встроенных типов — он задаёт лишь минимальные требования и соотношения между типами (`sizeof(char) == 1` всегда; `sizeof(short) <= sizeof(int) <= sizeof(long) <= sizeof(long long)`). Конкретные размеры зависят от платформы, ОС и компилятора — это implementation-defined behavior: в отличие от UB, это не "что угодно", а нечто конкретное, что обязана задокументировать каждая конкретная реализация.

## Типичные размеры int и char (x86_64, Linux, GCC)

На платформе x86_64 под Linux с компилятором GCC (модель данных LP64):

```cpp
sizeof(char);      // 1 байт  — гарантировано стандартом всегда
sizeof(short);     // 2 байта
sizeof(int);       // 4 байта
sizeof(long);      // 8 байт   (на Windows/MSVC long обычно 4 байта — модель LLP64)
sizeof(long long); // 8 байт
sizeof(float);     // 4 байта
sizeof(double);    // 8 байт
sizeof(void*);     // 8 байт   (отражает разрядность архитектуры)
```

`char` — единственный целочисленный тип, чей размер строго зафиксирован стандартом в 1 байт (это и есть определение байта в терминах C++, что не всегда означает ровно 8 бит, хотя на подавляющем большинстве современных платформ это так). Размер `long` — классическая ловушка: на x86_64 Linux/GCC это 8 байт, а на x86_64 Windows/MSVC — 4 байта из-за другой модели данных (LLP64 вместо LP64). Поэтому отвечая на вопрос "размер int" без уточнения платформы, нужно явно проговаривать, что это implementation-defined, и называть конкретную платформу как пример, а не как универсальный факт.

## Переполнение: unsigned vs signed

### Unsigned: переполнение строго определено

Арифметика `unsigned` определена стандартом как арифметика по модулю 2^n, где n — число бит типа. Поведение циклическое в обе стороны и гарантировано:

```cpp
unsigned char a = 255;
a = a + 1; // строго определено: 0      (255 + 1 = 256, mod 256 = 0)

unsigned char b = 0;
b = b - 1; // строго определено: 255    (0 - 1 = -1, mod 256 = 255)
```

### Signed: переполнение — undefined behavior

Для `signed` переполнение при арифметических операциях — UB, то есть стандарт не гарантирует никакого конкретного результата:

```cpp
int x = INT_MAX;
x = x + 1; // UB, а НЕ гарантированно INT_MIN
```

**Историческая причина.** До C++20 стандарт допускал три разных представления отрицательных чисел: дополнительный код (two's complement), обратный код (one's complement) и прямой код со знаковым битом (sign-magnitude). В разных представлениях "естественный" результат переполнения отличается бы — поэтому стандарт не мог зафиксировать конкретный результат, не привязываясь к конкретному представлению, и объявил переполнение signed неопределённым поведением целиком, а не просто implementation-defined. С C++20 стандарт зафиксировал two's complement как обязательное представление, но само переполнение при сложении/вычитании/умножении осталось UB — осознанное решение, а не недосмотр.

**Практическая причина.** Объявление переполнения signed как UB даёт компилятору пространство для оптимизаций. Классический пример — циклы:

```cpp
for (int i = 0; i <= n; i++) { ... }
```

Если бы переполнение `i` было определено (wrap-around), компилятору пришлось бы беспокоиться о крайнем случае `n == INT_MAX`, при котором `i++` дало бы `INT_MIN`, и условие `i <= n` снова стало бы истинным — бесконечный цикл. Раз переполнение signed — UB, компилятор вправе предположить, что такого не случится, и агрессивно оптимизировать цикл (векторизация, упрощение условий выхода) без генерации защитного кода на этот граничный случай.

### Сравнение

| | unsigned | signed |
|---|---|---|
| Поведение при переполнении | Гарантировано стандартом (mod 2^n) | UB |
| Можно полагаться в коде | Да | Нет |
| Связь с two's complement | Не имеет отношения — поведение задано напрямую | Совпадает с "наивной" битовой моделью two's complement, но не гарантировано языком |
