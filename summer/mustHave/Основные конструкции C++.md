# Основные конструкции C++

## static_cast

`static_cast` — оператор явного приведения типа, проверяемый на этапе компиляции. В отличие от C-style cast (`(int)x`), он не позволяет произвольное, потенциально опасное преобразование — компилятор проверяет, что преобразование вообще имеет смысл для данных типов.

```cpp
double d = 3.7;
int i = static_cast<int>(d); // явное преобразование double -> int, дробная часть отбрасывается

void* vp = &i;
int* ip = static_cast<int*>(vp); // явное преобразование void* -> int*

Base* b = new Derived();
Derived* d2 = static_cast<Derived*>(b); // приведение вниз по иерархии (без runtime-проверки!)
```

Синтаксис: `static_cast<ЦелевойТип>(выражение)`. Типичные случаи применения: явные сужающие числовые преобразования, преобразования между связанными типами указателей/ссылок в иерархии классов, преобразование `void*` обратно в конкретный тип указателя.

При преобразовании `void*` в `int*` физический адрес в памяти не меняется ни на бит — `void*` это просто "адрес без информации о типе", он намеренно не может быть разыменован напрямую. `static_cast<int*>(vp)` не трогает байты в памяти, а лишь возвращает указатель `int*` с тем же адресом — после этого компилятор знает, как трактовать байты по этому адресу при разыменовании (4-байтный `int`), и `*ip` становится допустимым.

Важно: `static_cast` не делает проверку в рантайме при downcast — если `b` на самом деле не указывает на `Derived`, поведение будет некорректным (в отличие от `dynamic_cast`).

## Пространства имён (namespace)

Группируют идентификаторы (функции, классы, переменные) под общим именем, чтобы избежать конфликтов имён между разными частями программы или библиотеками.

```cpp
namespace math {
    int square(int x) {
        return x * x;
    }
    const double PI = 3.14159;
}
```

Обращение к элементу внутри пространства имён — через оператор разрешения области видимости `::`:

```cpp
int result = math::square(5);
double pi = math::PI;
```

Чтобы не писать префикс каждый раз, используется `using`:

```cpp
using namespace math;   // делает видимыми ВСЕ имена из math без префикса
int result = square(5);

using math::square;     // делает видимым только square
int result2 = square(5);
double pi = PI;           // ошибка — PI не подключена через using
```

`using namespace` в заголовочных файлах считается плохой практикой, так как "протекает" во все файлы, подключающие этот заголовок, увеличивая риск конфликтов имён.

## auto

`auto` указывает компилятору вывести тип переменной из типа инициализирующего выражения на этапе компиляции — статическая типизация при этом сохраняется, тип фиксируется раз и навсегда, просто его не пишет программист:

```cpp
auto x = 5;       // int
auto y = 5.0;      // double
auto z = "hello";  // const char*
auto v = std::vector<int>{1, 2, 3}; // std::vector<int>
```

`auto` обязательно требует инициализатора:

```cpp
auto a; // ошибка компиляции — нечего выводить
```

По умолчанию `auto` отбрасывает ссылки и `const`/`volatile` верхнего уровня у инициализирующего выражения — выводится "как если бы" значение копировалось:

```cpp
const int& ref = someInt;
auto a = ref; // a — это int (НЕ const int&), просто копия значения
```

Чтобы сохранить ссылочную семантику или константность, нужно явно писать `auto&` или `const auto&`.

## range-based for

Синтаксис для итерации по всем элементам контейнера (или любого объекта с `begin()`/`end()`) без явной работы с итераторами:

```cpp
std::vector<int> v = {1, 2, 3, 4, 5};

for (int x : v) {
    std::cout << x; // x — копия каждого элемента
}
```

### auto в range-based for

```cpp
for (auto x : v) {        // x — копия элемента
    // ...
}

for (auto& x : v) {       // x — ссылка на элемент, без копирования, можно менять элемент
    x *= 2;                 // изменяет реальный элемент в v
}

for (const auto& x : v) { // x — константная ссылка, без копирования, менять нельзя
    std::cout << x;
    // x = 10; // ошибка компиляции
}
```

`auto x` копирует каждый элемент — для тяжёлых типов (`std::string`, `std::vector`, классы с большими полями) это лишние копии на каждой итерации. `auto& x` берёт ссылку на оригинальный элемент без копирования и позволяет модифицировать контейнер через `x`. `const auto& x` берёт константную ссылку без копирования и без права изменения — обычно предпочтительный вариант по умолчанию, когда элементы не нужно менять:

```cpp
std::vector<std::string> names = {"Alice", "Bob", "Charlie"};

for (const auto& name : names) { // нет копирования строк, нет случайной модификации
    std::cout << name << "\n";
}
```

## Синтаксис шаблонов: stack<T> поверх deque

Минимальный адаптер требует: шаблон класса с параметром-типом, поле — контейнер `std::deque<T>` внутри, методы, делегирующие вызовы в этот `deque`.

```cpp
#include <deque>

template<typename T>
class Stack {
public:
    void push(const T& value) {
        data_.push_back(value); // добавление в конец deque
    }

    void pop() {
        data_.pop_back(); // удаление с конца deque
    }

    T& top() {
        return data_.back(); // ссылка на последний элемент
    }

    const T& top() const {
        return data_.back(); // const-версия для константных объектов Stack
    }

    bool empty() const {
        return data_.empty();
    }

    size_t size() const {
        return data_.size();
    }

private:
    std::deque<T> data_; // внутреннее хранилище — все операции делегируются ему
};
```

Использование:

```cpp
Stack<int> s;
s.push(1);
s.push(2);
s.push(3);

std::cout << s.top();  // 3
s.pop();
std::cout << s.top();  // 2
std::cout << s.size(); // 2
```

Идея паттерна "адаптер" (как и реальный `std::stack`): класс не реализует собственную логику хранения, а оборачивает уже существующий контейнер, предоставляя ограниченный, специфичный интерфейс (`push`/`pop`/`top` — LIFO-доступ) поверх более общего интерфейса `deque` (произвольный доступ с обоих концов).

`template<typename T>` перед объявлением класса делает класс шаблонным; внутри тела `T` используется как обычный тип. Методы определены прямо внутри класса (inline) — для небольших шаблонных классов это обычно удобнее, чем выносить определения наружу.

# const

## const для переменных

Запрещает изменение значения после инициализации; переменная должна быть инициализирована при объявлении:

```cpp
const int x = 5;
x = 10; // ошибка компиляции
```

## const с указателями

`const` может относиться либо к тому, на что указывает указатель, либо к самому указателю — в зависимости от позиции относительно `*`.

```cpp
int x = 5, y = 10;

const int* p1 = &x;   // указатель на КОНСТАНТНЫЙ int
*p1 = 20;   // ошибка — нельзя менять значение через p1
p1 = &y;    // OK — можно перенаправить указатель

int* const p2 = &x;   // КОНСТАНТНЫЙ указатель на int
*p2 = 20;   // OK — меняем значение через p2
p2 = &y;    // ошибка — нельзя перенаправить p2

const int* const p3 = &x; // оба ограничения сразу
*p3 = 20;   // ошибка
p3 = &y;    // ошибка
```

Мнемоника: читать объявление справа налево относительно `*`. `const` слева от `*` (или перед типом: `const int* p`) — константно то, на что указывает указатель. `const` справа от `*` (`int* const p`) — константен сам указатель.

`const int* p` и `int const* p` — одно и то же; `const` можно писать как до, так и после типа.

## const со ссылками

Ссылка не может быть "перепривязана", поэтому есть только один осмысленный вариант — константность того, на что ссылаются:

```cpp
int x = 5;
const int& ref = x; // ссылка на константный int — через ref менять x нельзя

ref = 10;  // ошибка компиляции
x = 10;    // OK — напрямую через x менять можно
```

Только `const`-ссылка (или `&&`) может связываться с временным объектом (rvalue); продление времени жизни временного объекта (lifetime extension) работает именно для этого случая:

```cpp
const int& ref = 5; // создаётся временный int, время его жизни продлено до времени жизни ref
```

## const для функций-членов класса

`const` после списка параметров метода означает, что метод не изменяет состояние объекта, и его можно вызывать на константных объектах:

```cpp
class Box {
    int value;
public:
    int getValue() const {  // const-метод: гарантирует, что value не изменится
        return value;
    }
    void setValue(int v) {  // не const — может менять value
        value = v;
    }
};

const Box b{5};
b.getValue();   // OK — const-метод можно вызывать на const-объекте
b.setValue(10); // ошибка — нельзя вызвать не-const метод на const-объекте
```

Механически внутри `const`-метода неявный параметр `this` имеет тип `const Box*` вместо `Box*` — поэтому через `this`, а значит и через обращения к полям, ничего менять нельзя:

```cpp
class Box {
    int value;
public:
    int getValue() const {
        value = 100; // ошибка компиляции: this имеет тип const Box*
        return value;
    }
};
```

## const для параметров функций

Чаще всего применяется в сочетании со ссылкой — `const T&` — чтобы передать объект без копирования, но гарантировать, что функция его не изменит:

```cpp
void print(const std::string& s) { // без копирования строки, без права её менять
    std::cout << s;
    // s += "!"; // ошибка компиляции
}
```

Типичный паттерн для передачи "тяжёлых" объектов: избегаем дорогой копии (как при передаче по значению) и не даём функции изменить аргумент (как при обычной `T&`).

## Итог: зачем const

`const` — это контракт в коде, проверяемый компилятором: обещание, что значение/объект не будет изменено через данную ссылку/указатель/переменную/метод. Это позволяет ловить ошибки на этапе компиляции и делает код понятнее: сигнатура `void f(const std::string&)` сразу сообщает, что f не изменит переданную строку.
