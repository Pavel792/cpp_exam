# Термины и Undefined Behavior

## Свободная функция vs функция-член

Свободная функция (free function) объявлена вне класса, не привязана ни к какому объекту и не имеет неявного параметра `this`:

```cpp
int add(int a, int b) { return a + b; }
```

Функция-член (member function) объявлена внутри класса, имеет неявный параметр `this` — указатель на объект, для которого она вызвана, — и имеет доступ к приватным/защищённым членам класса:

```cpp
class Vector2D {
    double x, y;
public:
    double length() const { return std::sqrt(x*x + y*y); }  // this — Vector2D*
};
```

Статические методы класса — исключение: у них нет `this`, по сути это "свободные функции с областью видимости класса".

## Время жизни объекта

Время жизни (lifetime) объекта — период от завершения инициализации (конструктора) до начала уничтожения (деструктора). Категория хранения определяет, когда это происходит.

### Автоматическое (automatic storage duration)

Локальные переменные функций (не `static`, не `extern`). Создаются при входе в блок, уничтожаются при выходе из него — в порядке, обратном созданию:

```cpp
void f() {
    int x = 5;  // автоматическое
}  // x уничтожается здесь
```

### Динамическое (dynamic storage duration)

Объекты, созданные через `new`, живут в куче до явного `delete`. Программист сам управляет временем жизни — источник утечек памяти, double free, dangling-указателей:

```cpp
int* p = new int(5);  // динамическое
delete p;              // явное уничтожение
```

### Статическое (static storage duration)

Глобальные переменные, переменные в namespace scope, `static` внутри функций, `static`-члены класса. Создаются один раз, живут до завершения программы:

```cpp
void f() {
    static int counter = 0;  // инициализируется один раз
    counter++;
}
```

Отдельно стоит thread storage duration (`thread_local`) — свой экземпляр объекта на каждый поток.

## Undefined behavior

UB — поведение программы, для которого стандарт C++ не накладывает никаких ограничений. Компилятор может сгенерировать любой код: программа может упасть, дать неверный результат, "случайно" сработать правильно или вести себя непредсказуемо при разных запусках. Важно: UB — это не "гарантированный краш", а отсутствие гарантий вообще, в том числе гарантии стабильности — компилятор вправе предполагать, что UB не происходит, и оптимизировать код исходя из этого допущения.

### Неинициализированная переменная

```cpp
int x;
std::cout << x;  // UB — чтение неинициализированного значения
```

Говорить "там мусор, который иногда меняется" некорректно:

- компилятор может вообще не выделять для `x` память на стеке, а держать его в регистре или удалить выражение целиком;
- если переменная используется до инициализации, компилятор вправе предположить, что этот путь кода никогда не выполняется, и изменить или удалить связанный код — то есть непредсказуемо не только значение, но и само исполнение программы;
- слово "мусор" подразумевает детерминированность ("какой-то старый бит из памяти"), а значение может различаться при разных компиляциях, уровнях оптимизации и даже разных вызовах одной и той же функции.

Правильная формулировка: у переменной нет значения, потому что доступ к ней до инициализации — неопределённое поведение, и компилятор не обязан гарантировать ничего о результате выполнения такого кода.

### Выход за границы массива

```cpp
int arr[5] = {1, 2, 3, 4, 5};
int x = arr[10];  // UB
```

C++ не проверяет границы массивов в рантайме. `arr + 10` может указывать на произвольную область памяти — стек, другую переменную, невыделенную страницу. Возможные исходы: тихое чтение "случайного" значения, segfault, либо код, который работает на этапе разработки и ломается в продакшене при иных условиях компиляции.

### Dangling reference при возврате из функции

```cpp
int& f() {
    int local = 5;
    return local;  // UB: ссылка на объект, который сейчас будет уничтожен
}
```

`local` имеет автоматическое время жизни и уничтожается при выходе из `f()`. Возвращённая ссылка указывает на память, которая больше не принадлежит этому объекту — обращение к ней после возврата есть UB.

### Dangling reference при инвалидации в стандартном контейнере

```cpp
std::vector<int> v = {1, 2, 3};
int& ref = v[0];
v.push_back(4);  // может вызвать реаллокацию
ref = 10;        // UB, если реаллокация произошла
```

`vector` хранит элементы в непрерывном блоке памяти. Если текущая ёмкость (`capacity`) исчерпана, `push_back` выделяет новый, больший блок, перемещает туда элементы и освобождает старый. Все ссылки, указатели и итераторы на старые элементы становятся недействительными.

Разные операции инвалидируют по-разному:

```cpp
std::vector<int> v = {1, 2, 3};
v.push_back(4);   // инвалидирует только end(), если capacity хватило — ссылки целы
v.insert(v.begin(), 0);  // инвалидирует всё начиная с точки вставки
```

Для `std::list` и `std::map`/`std::set` ссылки и итераторы на не удалённые элементы остаются валидными даже при вставке/удалении других элементов — сильная гарантия по сравнению с `vector`.

### Разыменование nullptr

```cpp
int* p = nullptr;
int x = *p;  // UB
```

Это UB, а не гарантированный segfault, потому что стандарт не определяет содержимое "адреса 0" — это деталь платформы. На большинстве современных ОС нулевой адрес не замаплен в адресное пространство процесса, поэтому происходит segfault, но это поведение ОС, а не гарантия языка. На embedded-системах без MMU обращение по нулевому адресу может быть легитимным и тихо вернуть данные — отсюда и неопределённость: язык ничего не гарантирует.
