# Полиморфные классы

## Виртуальные методы и их отличия от невиртуальных

Невиртуальный метод вызывается на основе статического типа указателя/ссылки — типа, который компилятор видит на этапе компиляции:

```cpp
class Base {
public:
    void foo() { std::cout << "Base::foo\n"; } // не виртуальный
};

class Derived : public Base {
public:
    void foo() { std::cout << "Derived::foo\n"; } // скрывает, а не переопределяет
};

Base* b = new Derived();
b->foo(); // "Base::foo" — выбор по статическому типу указателя (Base*)
```

Виртуальный метод вызывается на основе динамического типа объекта — реального типа объекта в памяти, определяемого в рантайме:

```cpp
class Base {
public:
    virtual void foo() { std::cout << "Base::foo\n"; } // виртуальный
};

class Derived : public Base {
public:
    void foo() override { std::cout << "Derived::foo\n"; } // переопределяет
};

Base* b = new Derived();
b->foo(); // "Derived::foo" — выбор по динамическому типу объекта (Derived)
```

Механически это реализуется через виртуальную таблицу (vtable) — каждый объект полиморфного класса хранит скрытый указатель на таблицу указателей на функции, специфичную для его реального типа. Вызов виртуального метода — два разыменования (объект → vtable → нужная функция) вместо прямого вызова по адресу, известному во время компиляции, отсюда небольшие накладные расходы по сравнению с невиртуальными вызовами.

Это и есть суть полиморфизма времени выполнения: код в терминах `Base*` автоматически вызывает правильную реализацию метода в зависимости от реального объекта по этому указателю, без знания конкретного типа на этапе написания кода.

## Необходимость виртуального деструктора

Если класс используется полиморфно и объекты удаляются через `delete` базового указателя, деструктор базового класса обязан быть виртуальным:

```cpp
class Base {
public:
    ~Base() {} // не виртуальный — проблема
};

class Derived : public Base {
    int* data_;
public:
    Derived() : data_(new int[100]) {}
    ~Derived() { delete[] data_; }
};

Base* b = new Derived();
delete b; // UB: вызовется только ~Base(), ~Derived() не вызывается — утечка data_
```

Без `virtual` на деструкторе базового класса `delete b` вызывает деструктор только базового класса, основываясь на статическом типе указателя — точно так же, как обычный невиртуальный метод. Это явное UB по стандарту, не просто утечка.

```cpp
class Base {
public:
    virtual ~Base() {} // правильно
};

Base* b = new Derived();
delete b; // сначала ~Derived(), потом ~Base() — оба деструктора отрабатывают
```

Правило: если у класса есть хотя бы один виртуальный метод (класс задуман как полиморфная база), его деструктор должен быть виртуальным — даже если деструктору самому по себе нечего делать.

## Чисто виртуальные функции

Объявляются с `= 0` вместо тела и делают класс абстрактным — объект такого класса нельзя создать напрямую:

```cpp
class Shape {
public:
    virtual double area() const = 0; // чисто виртуальная функция
    virtual ~Shape() = default;
};

Shape s; // ошибка компиляции — нельзя создать объект абстрактного класса

class Circle : public Shape {
    double radius_;
public:
    Circle(double r) : radius_(r) {}
    double area() const override { return 3.14159 * radius_ * radius_; }
};

Circle c(5); // OK — Circle конкретный класс, переопределил area()
```

Класс остаётся абстрактным, пока хотя бы одна чисто виртуальная функция не переопределена во всех путях наследования.

### Что происходит при отсутствии override или virtual

`override` (C++11+) явно указывает компилятору: "этот метод обязан переопределять виртуальный метод базового класса". Без него легко допустить незаметную ошибку:

```cpp
class Base {
public:
    virtual void foo(int x) { std::cout << "Base::foo\n"; }
};

class Derived : public Base {
public:
    void foo(double x) { std::cout << "Derived::foo\n"; } // опечатка: double вместо int
    // без override компилятор молчит — это новая ПЕРЕГРУЗКА, а не переопределение
};

Base* b = new Derived();
b->foo(5); // "Base::foo" — ожидался полиморфный вызов Derived::foo, но сигнатуры не совпали
```

С `override` несоответствие сигнатур становится ошибкой компиляции вместо скрытого бага:

```cpp
class Derived : public Base {
public:
    void foo(double x) override { ... } // ошибка компиляции: не переопределяет ни один метод базы
};
```

Если забыть `virtual` в базовом классе вообще, "переопределение" в производном классе — это скрытие (hiding) метода базы, а не полиморфное переопределение: вызов через `Base*` всегда вызовет `Base::foo`, независимо от реального типа объекта.

## Slicing (срезка объектов)

Происходит при присваивании объекта производного класса переменной (не указателю/ссылке!) типа базового класса — копируется только часть, относящаяся к базовому классу, дополнительные поля производного класса теряются:

```cpp
class Base {
public:
    int baseField = 1;
    virtual void print() const { std::cout << "Base\n"; }
};

class Derived : public Base {
public:
    int derivedField = 2;
    void print() const override { std::cout << "Derived\n"; }
};

Derived d;
Base b = d; // slicing: b — объект типа Base, derivedField потерян безвозвратно

b.print(); // "Base" — не полиморфный вызов, b буквально объект типа Base
```

Slicing — не ошибка компиляции и не UB, это законная, но обычно нежелательная операция: `Base b = d` вызывает конструктор копирования `Base`, который принимает `const Base&` и не видит дополнительные поля `Derived`. Полиморфизм при этом полностью теряется.

### Как защититься от slicing

Никогда не хранить и не передавать полиморфные объекты по значению — только через указатель или ссылку:

```cpp
void process(const Base& obj) { // ссылка — нет slicing
    obj.print(); // полиморфный вызов работает корректно
}

void processBad(Base obj) { // по значению — slicing произойдёт при передаче Derived
    obj.print(); // всегда Base::print
}

Derived d;
process(d);    // OK
processBad(d); // slicing — derivedField и переопределение print() потеряны
```

Для владения объектами используются умные указатели — они тоже исключают slicing, потому что хранят указатель, а не сам объект:

```cpp
std::vector<std::unique_ptr<Base>> shapes;
shapes.push_back(std::make_unique<Derived>());
shapes[0]->print(); // полиморфный вызов — "Derived"
```

## Хранение полиморфных объектов в контейнерах

Прямое хранение по значению невозможно из-за slicing и из-за разного размера производных классов — `std::vector<Base>` физически не может хранить объекты `Derived` (они больше) в ячейках, рассчитанных под размер `Base`:

```cpp
std::vector<Base> shapes;
shapes.push_back(Derived()); // slicing при добавлении
```

Правильный подход — хранить указатели (предпочтительно умные) на базовый класс:

```cpp
std::vector<std::unique_ptr<Base>> shapes;
shapes.push_back(std::make_unique<Circle>(5));
shapes.push_back(std::make_unique<Square>(4));

for (const auto& shape : shapes) {
    std::cout << shape->area() << "\n"; // полиморфный вызов для каждого реального типа
}
```

`unique_ptr` (или `shared_ptr` для разделяемого владения) гарантирует, что каждый элемент вектора — указатель фиксированного размера, независимо от размера реального объекта, а сам объект выделен отдельно в куче и автоматически уничтожается через виртуальный деструктор.

## dynamic_cast для полиморфных классов

В отличие от `static_cast`, делает проверку в рантайме и требует, чтобы класс имел хотя бы один виртуальный метод.

### С указателями — nullptr при неудаче

```cpp
Base* b = new Derived();

Derived* d = dynamic_cast<Derived*>(b); // успех
if (d != nullptr) {
    d->derivedOnlyMethod();
}

Base* b2 = new Base();
Derived* d2 = dynamic_cast<Derived*>(b2); // неудача
if (d2 == nullptr) {
    std::cout << "приведение не удалось\n";
}
```

### Со ссылками — исключение std::bad_cast при неудаче

У ссылок нет аналога "пустого" значения вроде `nullptr`, поэтому при неудаче выбрасывается исключение:

```cpp
Base& bref = *b2;

try {
    Derived& dref = dynamic_cast<Derived&>(bref); // неудача
} catch (const std::bad_cast& e) {
    std::cout << "приведение не удалось: " << e.what() << "\n";
}
```

Механически `dynamic_cast` использует RTTI (runtime type information), хранящуюся вместе с vtable объекта — поэтому требуется полиморфный класс: без vtable неоткуда взять информацию о реальном типе объекта в рантайме.

# Перегрузка (overloading)

Возможность иметь несколько функций с одним именем, но разными сигнатурами (разным набором/типами параметров) в одной области видимости. Компилятор выбирает нужную версию на этапе компиляции, исходя из аргументов вызова:

```cpp
void print(int x) { std::cout << "int: " << x << "\n"; }
void print(double x) { std::cout << "double: " << x << "\n"; }
void print(const std::string& x) { std::cout << "string: " << x << "\n"; }

print(5);        // print(int)
print(3.14);      // print(double)
print("hello");   // print(const std::string&) — после неявной конвертации
```

## Это статический (compile-time) механизм

Выбор перегрузки происходит полностью на этапе компиляции, на основе типов аргументов в точке вызова. Это принципиально отличает перегрузку от виртуальных функций, где выбор реализации происходит в рантайме на основе динамического типа объекта:

```cpp
void foo(Base* b) {
    print(*b); // перегрузка print() выбирается ОДИН раз, по типу Base,
                // а не по реальному типу объекта, на который указывает b
}
```

## Перегрузка vs переопределение (override)

- **Перегрузка** — разные функции с одним именем, разными сигнатурами, в одной области видимости. Выбор на этапе компиляции.
- **Переопределение** — производный класс предоставляет новую реализацию виртуального метода базового класса с той же сигнатурой. Выбор в рантайме, по динамическому типу объекта.

```cpp
class Base {
public:
    virtual void foo(int x) {} // оригинал
};

class Derived : public Base {
public:
    void foo(int x) override {}  // переопределение — та же сигнатура
    void foo(double x) {}         // перегрузка — новая сигнатура, существует отдельно
};
```

Это напрямую связано с `override`: если в производном классе случайно изменить тип параметра у метода, который должен был переопределять виртуальный метод базы, компилятор без `override` молча создаст перегрузку вместо переопределения — отсюда баг с "не той" функцией при полиморфном вызове. `override` явно требует от компилятора: "это должно быть переопределение, иначе ошибка компиляции".

## Разрешение перегрузки (overload resolution)

Компилятор ищет наиболее подходящую перегрузку по ранжированию соответствия типов, упрощённо от лучшего к худшему: точное совпадение типа, совпадение с тривиальными преобразованиями (`int` → `const int`), продвижение типов (`char` → `int`, `float` → `double`), стандартные преобразования (`int` → `double`), пользовательские преобразования (через конструкторы/`operator`). Если после ранжирования остаётся несколько одинаково подходящих вариантов — это неоднозначность (ambiguous call), ошибка компиляции.
