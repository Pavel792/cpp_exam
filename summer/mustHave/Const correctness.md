# Const correctness

## Общая идея

Const correctness — дисциплина написания кода, при которой `const` расставляется максимально широко и последовательно везде, где объект/ссылка/указатель/метод не должен (и не будет) изменять состояние. Это не отдельная фича языка, а практика использования уже существующих механизмов (`const`-переменные, `const`-ссылки, `const`-методы), которая делает интерфейсы самодокументируемыми и позволяет компилятору ловить ошибки на этапе компиляции:

```cpp
class Vector2D {
public:
    double length() const { return std::sqrt(x_*x_ + y_*y_); } // не меняет объект — const
    void normalize() { /* меняет x_, y_ */ }                    // меняет объект — не const
private:
    double x_, y_;
};
```

Если код const-correct, сигнатуры функций сами сообщают, что они делают с данными, а компилятор гарантирует соблюдение этого контракта.

## Синтаксис константных ссылок при передаче аргументов

`const T&` — ссылка на константный объект: нельзя изменить через неё, но можно избежать копирования:

```cpp
void print(const std::string& s) { // стандартный паттерн для "тяжёлых" типов
    std::cout << s;
}
```

Правило по умолчанию для параметров функций: если функции не нужно изменять или владеть переданным объектом, параметр должен быть `const T&` (для нетривиальных типов) или просто `T` по значению (для дешёвых, тривиально копируемых типов вроде `int`, `double`):

```cpp
void f(int x);               // дешёвый тип — по значению, const не нужен
void g(const std::string& s); // тяжёлый тип, не модифицируется — const&
void h(std::string& s);       // тяжёлый тип, должен модифицироваться — обычная ссылка
void k(std::string s);        // тяжёлый тип, нужна собственная копия — по значению
```

Сигнатура `const T&` сразу сообщает вызывающему коду две вещи: функция не изменит переданный объект (контракт безопасности) и не сделает лишней копии (контракт производительности). Параметр `T&` без `const` сигнализирует, что функция планирует менять переданный объект.

## Невозможность изменять константные объекты и их поля

Если объект целиком объявлен `const`, любая попытка изменить любое его поле (в том числе через не-const методы) — ошибка компиляции:

```cpp
class Point {
public:
    Point(double x, double y) : x_(x), y_(y) {}
    void setX(double x) { x_ = x; }     // не const метод
    double getX() const { return x_; }  // const метод
private:
    double x_, y_;
};

const Point p(1.0, 2.0);
p.setX(5.0);          // ошибка компиляции: нельзя вызвать не-const метод на const объекте
double x = p.getX();  // OK — const метод можно вызывать на const объекте
```

Это распространяется и на прямой доступ к публичным полям:

```cpp
class Point {
public:
    double x, y;
};

const Point p{1.0, 2.0};
p.x = 5.0; // ошибка компиляции — нельзя менять поле const объекта напрямую
```

Компилятор обязан гарантировать, что объект, объявленный `const`, не изменится никаким путём за всё время своего существования — отсюда запрет на вызов неконстантных методов и на прямое присваивание полям.

### mutable

Иногда нужно разрешить изменение конкретного поля даже у `const`-объекта (например, кэширование внутри логически "константного" метода). Поле, помеченное `mutable`, можно изменять даже изнутри `const`-метода:

```cpp
class ExpensiveComputation {
    mutable bool cached_ = false;
    mutable double cachedValue_ = 0.0;
public:
    double getValue() const { // метод логически const — не меняет "видимое" состояние
        if (!cached_) {
            cachedValue_ = computeExpensive(); // OK благодаря mutable
            cached_ = true;
        }
        return cachedValue_;
    }
};
```

Это сохраняет логическую константность интерфейса (снаружи объект выглядит неизменным — повторные вызовы `getValue()` всегда дают тот же результат) при том, что физически внутри объекта что-то меняется (кэш заполняется один раз).

## const-qualifier у методов и перегрузка по const (const overloading)

`const` после списка параметров метода означает, что метод не изменяет состояние объекта (неявный `this` имеет тип `const T*`), и такой метод можно вызывать на `const`-объектах.

Можно объявить две версии одного метода — `const` и не-`const` — и компилятор выберет нужную в зависимости от константности объекта, на котором вызывается метод:

```cpp
class Buffer {
    std::vector<int> data_;
public:
    // неконстантная версия — вызывается на неконстантных объектах
    int& at(size_t i) {
        std::cout << "non-const version\n";
        return data_[i];
    }

    // константная версия — вызывается на константных объектах
    const int& at(size_t i) const {
        std::cout << "const version\n";
        return data_[i];
    }
};

Buffer b;
const Buffer cb;

b.at(0) = 5; // вызывает неконстантную версию — возвращает int&, можно присваивать
cb.at(0);     // вызывает константную версию — возвращает const int&, присвоить нельзя
```

`const` на методе считается частью его сигнатуры наравне с типами параметров, поэтому `at(size_t)` и `at(size_t) const` — два разных метода, и компилятор выбирает между ними так же, как при обычной перегрузке, но критерий выбора — не тип аргументов, а константность объекта, на котором вызывается метод.

Неконстантная версия может возвращать неконстантную ссылку (`int&`), позволяя читать и изменять элемент, а константная версия возвращает `const int&` — тот же интерфейс безопасно работает и с константными, и с неконстантными объектами, давая ровно те права доступа, которые уместны в каждом случае. Так устроены методы `operator[]`, `at()`, `front()`, `back()` у стандартных контейнеров — у каждого есть обе версии:

```cpp
std::vector<int> v = {1, 2, 3};
const std::vector<int> cv = {1, 2, 3};

v[0] = 100;     // OK — неконстантная operator[] возвращает int&
// cv[0] = 100; // ошибка — константная operator[] возвращает const int&
int x = cv[0];  // OK — чтение через const-версию разрешено
```
