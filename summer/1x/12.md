# 12. Теоретическая и практическая всячина

## Возвращаемый тип `auto`

### Простой вывод типа

```cpp
auto add(int a, int b) {
    return a + b;  // компилятор выводит int
}
```

Компилятор смотрит на `return` и выводит тип. Правила те же что и для `auto` переменных — снимаются ссылки и `const`:

```cpp
int x = 5;
auto foo() { return x; }   // возвращает int, не int&
auto& bar() { return x; }  // возвращает int&
```

Если несколько веток — все должны возвращать одинаковый тип:

```cpp
// ✅ обе ветки возвращают int
auto lam = [](int x) {
    if (x > 0) return x;
    return 0;
};

// ❌ разные типы — компилятор не знает что выбрать
auto lam = [](int x) {
    if (x > 0) return x;    // int
    return 0.5;              // double
};
```

### Рекурсия

Компилятор должен увидеть `return` с конкретным типом до рекурсивного вызова:

```cpp
auto factorial(int n) {
    if (n <= 1) return 1;           // ✅ тип известен здесь
    return n * factorial(n - 1);    // ✅ тип уже выведен
}
```

### Trailing return type: `auto` + `->`

Удобно когда тип зависит от параметров, или при определении метода снаружи класса:

```cpp
// тип зависит от параметров
template<typename A, typename B>
auto add(A a, B b) -> decltype(a + b) {
    return a + b;
}

// определение метода снаружи класса
struct MyClass {
    using Iterator = int*;
    Iterator begin();
};

// без trailing — нужно квалифицировать тип
MyClass::Iterator MyClass::begin() { return data; }

// с trailing — Iterator уже в скоупе MyClass
auto MyClass::begin() -> Iterator { return data; }
```

### Объявление функций с `auto`

Можно объявить без определения только с trailing return type:

```cpp
auto foo() -> int;   // ✅ тип указан явно
auto foo();          // ❌ без определения компилятор не знает тип
```

### Явное указание возвращаемого типа лямбды

```cpp
// нужен явный тип когда ветки возвращают разные типы
auto lam = [](int x) -> double {
    if (x > 0) return x;   // int приводится к double
    return 0.5;
};
```

## Оператор `switch`

### `break` и `[[fallthrough]]`

Без `break` выполнение проваливается в следующий `case`:

```cpp
switch (x) {
    case 1:
        foo();
        // нет break — провалится в case 2!
    case 2:
        bar();
        break;
}
```

Если провал намеренный — пишем `[[fallthrough]]`, иначе компилятор предупредит (`-Wimplicit-fallthrough`):

```cpp
switch (x) {
    case 1:
        foo();
        [[fallthrough]];  // явно говорим: провал намеренный
    case 2:
        bar();
        break;
}
```

### Проблемы с инициализацией переменных внутри `switch`

Нельзя объявить переменную в одном `case` — она видна в других, но инициализация может быть пропущена:

```cpp
switch (x) {
    case 1:
        int y = 5;  // ❌ ошибка компиляции
        break;
    case 2:
        y = 10;     // y виден, но мог не инициализироваться
}
```

Решение — обернуть в блок `{}`:

```cpp
switch (x) {
    case 1: {
        int y = 5;  // ✅ y живёт только внутри блока
        foo(y);
        break;
    }
    case 2:
        bar();
        break;
}
```

## Глобальные переменные, `extern`, `inline`, static initialization order fiasco

### `extern`

```cpp
// file1.cpp
int x = 42;      // определение

// file2.cpp
extern int x;    // объявление — x определён где-то ещё
int y = x;
```

### Static initialization order fiasco

Порядок инициализации глобальных переменных между разными `.cpp` не определён:

```cpp
// file1.cpp
int x = 42;

// file2.cpp
extern int x;
int y = x;  // ❌ x может быть ещё не инициализирован!
```

Внутри одного файла — порядок гарантирован сверху вниз.

### `inline` переменные и функции

Без `inline` определение в заголовке нарушает ODR если заголовок подключён в несколько `.cpp`:

```cpp
// header.h
int x = 42;       // ❌ ODR нарушение
inline int y = 42; // ✅ одна копия на всю программу
```

Методы класса определённые внутри класса — автоматически `inline`. Снаружи — нужно явно:

```cpp
struct Foo {
    void bar() { }           // автоматически inline
    void baz();
};

inline void Foo::baz() { }   // ✅ можно в заголовке
// void Foo::baz() { }       // ❌ ODR если заголовок в нескольких .cpp
```

### `inline` vs `static` vs unnamed namespace

`inline` — одна копия, shared между всеми файлами.
`static` и unnamed namespace — каждый `.cpp` получает свою копию.

```cpp
// header.h
inline int x = 42;   // одна переменная на всю программу

// file.cpp
static int y = 42;   // своя копия в этом .cpp
namespace {
    int z = 42;      // тоже своя копия — но работает и для типов
}
```

Unnamed namespace умеет больше чем `static` — работает для типов:

```cpp
namespace {
    struct Bar {};   // ✅
}
static struct Baz{}; // ❌ static не применяется к типам
```

| | Одна копия | Можно в заголовке | Работает для типов |
|---|---|---|---|
| `inline` | ✅ | ✅ | — |
| `static` | ❌ | ✅ | ❌ |
| unnamed namespace | ❌ | ✅ | ✅ |

## `constexpr` vs `const` для констант

### Разница

```cpp
const int x = foo();      // runtime константа — вычисляется при запуске
constexpr int y = 42;     // compile-time константа — известна при компиляции
```

### Порядок инициализации

`constexpr` решает static initialization order fiasco — инициализируется до любого runtime кода:

```cpp
// file1.cpp
constexpr int x = 42;  // compile-time — всегда готов

// file2.cpp
extern const int x;
int y = x;  // ✅ x гарантированно инициализирован
```

### Статические константы-члены класса

```cpp
struct Foo {
    static const int x = 42;          // ✅ только для целых типов
    static constexpr double y = 3.14;  // ✅ для любых типов
    static const double z = 3.14;      // ❌ не целый тип
};
```

### `inline constexpr` и множественное определение

```cpp
// header.h
constexpr int x = 42;        // неявно static — своя копия в каждом .cpp
inline constexpr int y = 42; // одна копия на всю программу
```

Если берёшь адрес константы — без `inline` адреса в разных `.cpp` будут разными:

```cpp
// file1.cpp
const int* p1 = &x;  // адрес своей копии

// file2.cpp
const int* p2 = &x;  // адрес другой копии

p1 == p2;  // false без inline, true с inline
```

Итоговое правило для глобальных констант в заголовке:

| | Compile-time | Можно в заголовке | Одна копия |
|---|---|---|---|
| `const` | ❌ | ❌ (ODR) | ❌ |
| `constexpr` | ✅ | ✅ (неявно static) | ❌ |
| `inline constexpr` | ✅ | ✅ | ✅ |

Современный C++: глобальные константы объявляй как `inline constexpr` в заголовке.
