# 61. Виртуальное наследование

## Проблема ромба и виртуальный базовый класс

```
    A
   / \
  B   C
   \ /
    D
```

Без виртуального наследования — два подобъекта `A`:

```cpp
#include <iostream>

struct A { int x = 42; };
struct B : A {};
struct C : A {};
struct D : B, C {};  // два A!

int main() {
    D obj;
    // obj.x;       // ошибка — неоднозначность
    obj.B::x = 1;   // независимые копии
    obj.C::x = 2;
    // A* pa = &obj; // ошибка — какой A?
}
```

С виртуальным наследованием — один `A`:

```cpp
struct B : virtual A {};
struct C : virtual A {};
struct D : B, C {};  // один A!

int main() {
    D obj;
    obj.x = 42;      // ok — один A
    A* pa = &obj;    // ok — однозначно
}
```

---

## Порядок инициализации и уничтожения

1. **Виртуальные базы** — самым производным классом, первыми
2. **Невиртуальные базы** — в порядке объявления
3. **Поля** — в порядке объявления
4. **Тело конструктора**

Уничтожение — строго обратный порядок.

```cpp
#include <iostream>

struct A {
    int x;
    A(int x) : x(x) { std::cout << "A(" << x << ")\n"; }
    ~A() { std::cout << "~A\n"; }
};

struct B : virtual A {
    B() : A(1) { std::cout << "B()\n"; }  // A(1) игнорируется при создании D!
    ~B() { std::cout << "~B\n"; }
};

struct C : virtual A {
    C() : A(2) { std::cout << "C()\n"; }  // A(2) игнорируется при создании D!
    ~C() { std::cout << "~C\n"; }
};

struct D : B, C {
    D() : A(42), B(), C() { std::cout << "D()\n"; }  // D обязан инициализировать A!
    ~D() { std::cout << "~D\n"; }
};

int main() {
    D obj;   // A(42) → B() → C() → D()
             // ~D → ~C → ~B → ~A

    B b;     // A(1) → B()  ← B сам инициализирует A когда он финальный класс
}
```

**Ключевое правило:** виртуальную базу инициализирует **самый производный класс**. `B` и `C` тоже пишут `A(...)` в списке инициализации — но при создании `D` эти вызовы игнорируются.

## Представление в памяти (неполиморфные классы)

Виртуальная база всегда **в конце** — её позиция зависит от финального класса, компилятор не знает заранее где она будет:

```cpp
struct A { int x = 42; };
struct B : virtual A { int b = 1; };
struct C : virtual A { int c = 2; };
struct D : B, C { int d = 3; };
```

```
D в памяти (адреса условные):
адрес:  0        4   8        12  16  20
       [ vptr_B | b ][ vptr_C | c ][ d ][ x ]
           ↓               ↓
       таблица_B        таблица_C
       [offset=+16]     [offset=+8]
```

`vptr_B` → таблица смещений → "A находится на +16 от начала B-подобъекта"
`vptr_C` → таблица смещений → "A находится на +8 от начала C-подобъекта"

Оба смещения ведут на один `A` по адресу 20.

### Почему нужен vptr в каждой виртуальной базе

`B` не знает в каком финальном классе будет жить — смещение до `A` разное:

```
просто B:  [ vptr_B | b | A ]        — смещение до A = +8
D:         [ vptr_B | b | vptr_C | c | d | A ]  — смещение до A = +16
```

Смещение хранится в таблице, `vptr` указывает на неё. При создании `D` компилятор записывает правильные смещения.

## Виртуальное и невиртуальное одновременно

```cpp
#include <iostream>

struct A { int x = 42; };
struct B : virtual A {};  // виртуально — A в конце, по указателю
struct C : A {};          // невиртуально — своя копия A внутри C

struct D : B, C {};       // два A! один виртуальный, один в C

int main() {
    D obj;
    obj.B::x;  // виртуальный A — в конце
    obj.C::x;  // невиртуальный A — внутри C подобъекта
    // obj.x;  // ошибка — неоднозначность
}
```

```
D в памяти:
[ B[vptr|поля B] | C[A внутри C | поля C] | поля D | A(виртуальный) ]
                        ↑                                ↑
                  невиртуальный A               виртуальный A
```

- **Невиртуальный** — `A` физически внутри подобъекта, прямой доступ
- **Виртуальный** — `A` в конце, доступ через vptr и смещение

## `dynamic_cast` с виртуальными методами и без

### С виртуальными методами — работает

RTTI хранится в vtable — нужна хотя бы одна виртуальная функция:

```cpp
#include <iostream>

struct A { virtual ~A() = default; };  // виртуальный деструктор — RTTI есть
struct B : virtual A {};
struct C : virtual A {};
struct D : B, C {};

int main() {
    D obj;
    B* pb = &obj;

    C* pc = dynamic_cast<C*>(pb);   // cross-cast — ok
    D* pd = dynamic_cast<D*>(pb);   // вниз — ok
    A* pa = dynamic_cast<A*>(pb);   // вверх — ok

    std::cout << (pc ? "успех" : "nullptr") << "\n";
}
```

### Без виртуальных методов — невозможно

```cpp
struct A { };  // нет vtable — нет RTTI
struct B : virtual A {};

int main() {
    B obj;
    A* pa = &obj;
    B* pb = dynamic_cast<B*>(pa);  // ошибка компиляции!
    // "source type is not polymorphic"
}
```

Цепочка: виртуальные функции → vtable → `type_info` в vtable → RTTI → `dynamic_cast`.

## Delegate-to-sister

Один базовый класс вызывает метод **другого базового класса** через виртуальную диспетчеризацию — не зная о его существовании:

```cpp
#include <iostream>

struct A {
    virtual void foo() { std::cout << "A::foo\n"; }
    virtual void bar() { std::cout << "A::bar\n"; }
};

struct B : virtual A {
    void foo() override {
        std::cout << "B::foo\n";
        bar();  // чей bar? зависит от финального класса!
    }
};

struct C : virtual A {
    void bar() override { std::cout << "C::bar\n"; }
};

struct D : B, C {};

int main() {
    D obj;
    obj.foo();
    // B::foo
    // C::bar  ← B делегировал к сестре C!

    B b;
    b.foo();
    // B::foo
    // A::bar  ← просто B, C нет — bar из A
}
```

### Почему работает

Компилятор при создании `D` строит vtable для каждого подобъекта:
- `foo` переопределил только `B` → `B::foo`
- `bar` переопределил только `C` → `C::bar`

```
vtable "B внутри D":  [ B::foo | C::bar ]  ← компилятор записал C::bar
vtable "B само по себе": [ B::foo | A::bar ]
```

Когда `B::foo` вызывает `bar()` через `this` — читает vtable финального объекта `D` — там `C::bar`.

Если оба переопределили одно и то же — **ошибка компиляции**, неоднозначность. `D` должен сам переопределить:

```cpp
struct D : B, C {
    void foo() override { B::foo(); }  // D решает явно
};
```

## Невозможность `static_cast` к наследнику от виртуальной базы

`static_cast` работает на этапе компиляции — зашивает смещение константой. С виртуальной базой смещение зависит от финального класса — неизвестно на этапе компиляции:

```cpp
struct A { int x; };
struct B : virtual A { int y; };
struct D : B { int z; };

/*
просто B:  [ vptr | y | A ]      — смещение от A до B = -8
D:         [ vptr | y | z | A ]  — смещение от A до B = -12
*/

int main() {
    D obj;
    A* pa = &obj;
    // B* pb = static_cast<B*>(pa);  // ошибка компиляции!
    // компилятор видит только A* — не знает реальный тип

    // только dynamic_cast — вычисляет смещение в рантайме через RTTI
    struct A2 { virtual ~A2() = default; };
    struct B2 : virtual A2 {};
    struct D2 : B2 {};

    D2 obj2;
    A2* pa2 = &obj2;
    B2* pb2 = dynamic_cast<B2*>(pa2);  // ok
    std::cout << (pb2 ? "успех" : "nullptr") << "\n";
}
```

## Касты при виртуальном наследовании

| Каст | Направление | Виртуальная база |
|---|---|---|
| Неявный | только вверх | ok |
| `static_cast` | вверх | ok |
| `static_cast` | вниз от виртуальной базы | ❌ ошибка компиляции |
| `dynamic_cast` | вверх/вниз/cross | ok — нужен полиморфный тип |

```cpp
struct A { virtual ~A() = default; };
struct B : virtual A {};
struct D : B {};

int main() {
    D obj;

    A* pa = &obj;              // неявно вверх — ok
    B* pb = &obj;              // неявно вверх — ok

    // static_cast вниз от виртуальной базы — нельзя
    // B* pb2 = static_cast<B*>(pa);  // ошибка!

    // dynamic_cast — можно
    B* pb3 = dynamic_cast<B*>(pa);  // ok
    D* pd  = dynamic_cast<D*>(pa);  // ok
}
```
