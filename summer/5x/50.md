# 50. Trivially Copyable, Padding, Standard Layout, POD

## Кодировки строк и порядок байт

### Кодировки

```cpp
#include <iostream>
#include <cstring>

int main() {
    const char* ascii = "Hello";    // 1 байт на символ
    const char* utf8  = "Привет";   // кириллица = 2 байта на символ

    std::cout << strlen(ascii) << "\n";  // 5
    std::cout << strlen(utf8)  << "\n";  // 12, не 6!
}
```

При передаче данных между системами — нужно договориться о кодировке заранее.

### Порядок байт (endianness)

```cpp
#include <cstdint>
#include <cstdio>

int main() {
    uint32_t x = 0x12345678;
    uint8_t* bytes = reinterpret_cast<uint8_t*>(&x);

    // на little-endian (x86):
    printf("%02X %02X %02X %02X\n", bytes[0], bytes[1], bytes[2], bytes[3]);
    // выведет: 78 56 34 12 — младший байт первый
}
```

```
Big-endian    (сеть, Motorola): 12 34 56 78  — старший байт первый
Little-endian (x86, ARM):       78 56 34 12  — младший байт первый
```

При передаче чисел по сети — нужно конвертировать порядок байт (`htonl`, `ntohl`).

## `reinterpret_cast` и Strict Aliasing Rule (SAR)

SAR — часть стандарта C++. Компилятор предполагает что указатели **разных типов** не указывают на одну память. Это даёт право агрессивно оптимизировать — кэшировать значения, не перечитывать из памяти.

Нарушение — читать память через указатель чужого типа:

```cpp
int x = 42;
float* f = reinterpret_cast<float*>(&x);
float val = *f;  // UB! компилятор может подсунуть закэшированное значение
```

Компилятор видит `int*` и `float*` — считает что они не пересекаются — может не перечитать актуальные байты.

### Корректное преобразование — `memcpy`

`memcpy` — просто копирует N байт без интерпретации типов. Не нарушает SAR:

```cpp
#include <cstring>
#include <cstdint>
#include <cstdio>

int main() {
    float f = 3.14f;
    uint32_t bits;

    // корректно — создаём свою копию байт
    memcpy(&bits, &f, sizeof(bits));
    printf("float bits: %08X\n", bits);

    // обратно
    float f2;
    memcpy(&f2, &bits, sizeof(f2));
    printf("%.2f\n", f2);  // 3.14
}
```

Исключения из SAR — ими можно читать любую память:
- `char*`, `unsigned char*`, `std::byte*`
- `memcpy`

```cpp
int x = 42;
// легально — char* может смотреть на любую память
char* bytes = reinterpret_cast<char*>(&x);
for (int i = 0; i < 4; i++)
    printf("%02X ", (unsigned char)bytes[i]);  // ok!
```

**Золотое правило:** хочешь посмотреть на байты чужого типа — сделай копию через `memcpy`, не разыменовывай чужой указатель.

## Trivially Copyable

Структуру можно корректно скопировать просто копированием байт (`memcpy`). Никакой логики в конструкторах/деструкторах.

### Требования

- Нет пользовательского конструктора копирования/перемещения
- Нет пользовательского оператора присваивания
- Нет пользовательского деструктора
- Все поля тоже Trivially Copyable

```cpp
#include <iostream>
#include <type_traits>

struct Point {           // Trivially Copyable — просто данные
    float x, y, z;
};

struct NotTrivial {      // НЕ Trivially Copyable — логика в деструкторе
    int* data;
    NotTrivial() : data(new int(42)) {}
    ~NotTrivial() { delete data; }
};

int main() {
    std::cout << std::is_trivially_copyable_v<Point>      << "\n";  // 1
    std::cout << std::is_trivially_copyable_v<NotTrivial> << "\n";  // 0
}
```

### Нетривиально копируемые типы

`std::string`, `std::vector`, `std::unique_ptr`, `std::shared_ptr` — хранят указатели на кучу. `memcpy` скопирует только указатель, не данные:

```cpp
std::vector<int> v1 = {1, 2, 3};
std::vector<int> v2;
memcpy(&v2, &v1, sizeof(v1));  // v1 и v2 указывают на одну память!
// ~v1 → delete[] ptr  ← ok
// ~v2 → delete[] ptr  ← UB! двойное удаление
```

### Использование для сериализации

```cpp
#include <cstring>
#include <cstdint>
#include <iostream>

struct Packet {
    uint32_t id;
    float x;
    float y;
};

int main() {
    Packet p{1, 3.14f, 2.71f};

    // сериализация — просто копируем байты
    char buffer[sizeof(Packet)];
    memcpy(buffer, &p, sizeof(Packet));

    // десериализация
    Packet p2;
    memcpy(&p2, buffer, sizeof(Packet));
    std::cout << p2.id << " " << p2.x << " " << p2.y << "\n";
}
```

## Padding (выравнивание)

Процессор читает память выровненными блоками — `int` по адресу кратному 4, `double` — кратному 8. Компилятор добавляет пустые байты (padding) между полями:

```cpp
#include <iostream>

struct Bad {
    char a;   // 1 байт
              // 3 байта padding
    int b;    // 4 байта
    char c;   // 1 байт
              // 3 байта padding
};            // итого 12 байт!

struct Good {
    int b;    // 4 байта
    char a;   // 1 байт
    char c;   // 1 байт
              // 2 байта padding
};            // итого 8 байт

int main() {
    std::cout << sizeof(Bad)  << "\n";  // 12
    std::cout << sizeof(Good) << "\n";  // 8
}
```

**Правило:** располагай поля от большего к меньшему — padding минимален.

### Отключение padding — `#pragma pack`

```cpp
#pragma pack(push, 1)
struct Packet {
    char a;   // 1 байт
    int b;    // 4 байта — прямо после a
    char c;   // 1 байт
};            // итого 6 байт
#pragma pack(pop)

int main() {
    std::cout << sizeof(Packet) << "\n";  // 6
}
```

### Последствия невыровненных типов

- **Примитивные типы** (`int`, `float`) — на x86 просто медленнее (два обращения к памяти), на ARM — крэш
- **Указатели** — невыровненный указатель читается неправильно → мусорный адрес → крэш
- **Нетривиальные типы** (`std::vector`, `std::string`) — содержат указатели → UB

```cpp
#pragma pack(push, 1)
struct Bad {
    char a;
    std::vector<int> v;  // вектор по невыровненному адресу — UB!
};
#pragma pack(pop)
```

### Ссылки и указатели на невыровненную память

```cpp
#pragma pack(push, 1)
struct Packed {
    char a;
    int b;   // адрес +1, не выровнен
    int c;   // адрес +5, не выровнен
};
#pragma pack(pop)

// плохо — компилятор генерирует выровненное чтение int& → UB на ARM
void swap_bad(Packed& p) {
    int tmp = p.b;
    p.b = p.c;
    p.c = tmp;
}

// хорошо — memcpy безопасно читает невыровненную память
void swap_good(Packed& p) {
    int tmp_b, tmp_c;
    memcpy(&tmp_b, &p.b, sizeof(int));
    memcpy(&tmp_c, &p.c, sizeof(int));
    memcpy(&p.b, &tmp_c, sizeof(int));
    memcpy(&p.c, &tmp_b, sizeof(int));
}
```

## Standard Layout

Структура совместима с C по расположению в памяти. Поля лежат в том же порядке что объявлены.

### Требования и почему они важны

- **Нет виртуальных функций** — компилятор добавляет скрытый `vptr` в начало, C не знает о нём
- **Все поля одного уровня доступа** — компилятор может переставлять поля с разным доступом
- **Нет полей и в базовом и в наследнике** — непонятно где граница между ними в памяти

```cpp
#include <type_traits>
#include <iostream>

struct Good { float x; float y; };          // Standard Layout

struct Bad1 { virtual void foo() {} int x; };  // vptr — не SL

struct Bad2 {                               // разный доступ — не SL
    public:  int x;
    private: int y;
};

int main() {
    std::cout << std::is_standard_layout_v<Good> << "\n";  // 1
    std::cout << std::is_standard_layout_v<Bad1> << "\n";  // 0
    std::cout << std::is_standard_layout_v<Bad2> << "\n";  // 0
}
```

## POD (Plain Old Data)

POD = **Trivially Copyable** + **Standard Layout**. Полностью совместима с C.

```cpp
#include <type_traits>
#include <iostream>

struct POD { int x; float y; char z; };

struct NotPOD1 { int x; std::string s; };   // не trivially copyable
struct NotPOD2 { virtual void foo() {} int x; };  // не standard layout

int main() {
    std::cout << std::is_trivially_copyable_v<POD> << "\n";  // 1
    std::cout << std::is_standard_layout_v<POD>    << "\n";  // 1
    // в C++20 is_pod удалён — проверяем оба условия отдельно
}
```
